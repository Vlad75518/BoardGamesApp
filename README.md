# 🎲 Симулятор Настільних Ігор (Board Games Simulator)

## 📌 Опис проекту
Цей проект є лабораторною роботою з дисципліни **«Архітектура та проектування програмного забезпечення»** (Варіант 14). Застосунок являє собою об'єктно-орієнтований консольний симулятор настільних ігор (**Шахи, Шашки, Монополія, Нарди**), розроблений на платформі .NET (C#).

Головна мета проекту — продемонструвати практичне застосування фундаментальних архітектурних паттернів, принципів проектування класу **SOLID**, а також концепцій **KISS, YAGNI, DRY**, високої зв'язності логіки (**High Cohesion**) та низької залежності компонентів (**Low Coupling**).

---

## 🏗️ Архітектура проекту та дотримання принципів проектування

Проект спроектовано таким чином, щоб повністю відокремити бізнес-логіку предметної області від інтерфейсу користувача (консольного відображення).

### 1. Дотримання принципів SOLID:
* **Single Responsibility Principle (SRP) / Принцип єдиної відповідальності:** Кожен клас виконує лише одну чітко визначену функцію. Наприклад, класи ігор (`ChessGame`, `MonopolyGame`) відповідають лише за правила та конфігурацію конкретної гри, класи дій (`MoveAction`, `RollDiceAction`) описують виключно поведінку ходу, а `GameEngine` керує лише життєвим циклом та симуляцією ігрового процесу.
* **Open/Closed Principle (OCP) / Принцип відкритості/закритості:** Архітектура відкрита для розширення, але закрита для змін. Додавання нової гри (наприклад, *Uno*) або нової дії (наприклад, *DrawCardAction*) відбувається шляхом створення нових класів, які наслідують абстракції, без необхідності модифікувати існуючий ігровий рушій `GameEngine` чи інші компоненти.
* **Liskov Substitution Principle (LSP) / Принцип підстановки Лісков:** Усі конкретні класи ігор (наприклад, `CheckersGame`) є повноцінними спадкоємцями абстрактного класу `BoardGame`. Рушій `GameEngine` працює суто з базовим класом `BoardGame`, і будь-яка гра може бути підставлена у рушій без зміни коректності роботи програми.
* **Interface Segregation Principle (ISP) / Принцип розділення інтерфейсу:** Інтерфейс дій `IGameAction` є максимально простим, компактним і містить лише необхідні для симуляції члени (`Description` та метод `Execute`), що запобігає примусовому впровадженню зайвих методів.
* **Dependency Inversion Principle (DIP) / Принцип інверсії залежностей:** Високорівневі модулі (`GameEngine`, `Program`) не залежать від низькорівневих деталей (конкретних ігор чи дій). Вони взаємодіють через абстрактний клас `BoardGame` та інтерфейс `IGameAction`.

### 2. Принципи KISS, YAGNI та DRY:
* **KISS (Keep It Simple, Stupid):** Замість створення складних перевантажених сервісів керування чергою гравців (`TurnManager`), логіка переходу ходу реалізована за допомогою лаконічної математичної формули кільцевого буфера: `currentPlayerIndex = (currentPlayerIndex + 1) % game.Players.Count;`. Це гарантує простоту та надійність перемикання гравців.
* **YAGNI (You Aren't Gonna Need It):** В архітектурі немає надлишкового коду. Класи фігур `Checker` та `ChessFigure` залишені порожніми спадкоємцями `Piece` свідомо — для майбутнього візуального розширення, тоді як повторювана валідація була успішно об'єднана.
* **DRY (Don't Repeat Yourself):** Загальна логіка перевірки можливості старту гри та виконання кроків винесена в базовий клас `BoardGame`, що виключає дублювання коду в окремих іграх.

### 3. Зв'язність та Залежність (Cohesion & Coupling):
* **High Cohesion (Висока внутрішня зв'язність):** Кожен модуль містить компоненти, які логічно пов'язані між собою (наприклад, папка `Models/Actions` містить суто атомарні ігрові дії).
* **Low Coupling (Низька взаємозалежність компонентів):** Реалізовано через **механізм подій (Events)**. Класи ігор нічого не знають про консоль і не містять викликів `Console.WriteLine`. Замість цього вони генерують події (`TurnMade`, `PlayerWon`), на які підписується зовнішній шар програми. Це дозволяє за потреби легко замінити консольний UI на графічний (Web/Desktop) без жодних змін у логіці ігор.

## 🕹️ Логіка симуляції ("Ідеальний світ")
Згідно з вимогами до лабораторної роботи, застосунок реалізує модель «ідеального світу»:

Ігри запускаються миттєво і завжди готові до процесу.

Гравці роблять ходи у чіткій циклічній послідовності.

Оскільки застосунок є архітектурним тренажером, а не повноцінною грою, на кожному кроці для гравця обирається випадкова дозволена дія зі списку тих, що закладені в правила конкретної гри через GameRules.AllowedActions.

Наприклад: У Монополії гравець випадково викине кубик, купить, продасть нерухомість або сплатить оренду, тоді як у Нардах він симулюватиме лише кидок кубика, хід або виведення фішки.

Умова завершення гри: На кожному кроці симулюється випадкова ймовірність завершення партії (шанс 15%). Якщо рандом спрацьовує — поточний гравець оголошеється переможцем, викликається подія PlayerWon, і цикл завершується. Це дозволяє уникнути нескінченних циклів флуду в консолі та наочно демонструє роботу архітектурних зв'язків.

## 💻 Інтерфейс користувача (Console UI)
Консольний інтерфейс було оновлено для покращення презентаційного вигляду під час демонстрації:

Безшовне меню: Програма працює у безкінечному циклі в Program.cs. Після завершення будь-якої партії користувач може натиснути будь-яку клавішу, екран очиститься (Console.Clear()), і він повернеться до головного меню вибору ігор без необхідності перезапускати застосунок через F5.

Кольорове кодування:

Зелений колір — старт та завершення ігрової сесії.

Червоний колір — критичні помилки валідації або вихід.

Циановий + Білий — логування поточних ходів гравців.

Пурпуровий (Magenta) — урочисте оголошення переможця гри.

Часова затримка: Завдяки використанню Thread.Sleep(...) лог ходів з'являється плавно (з інтервалом у 800 мс), що дозволяє комфортно спостерігати за розвитком симуляції в реальному часі.

## 🚀 Як запустити проект
Відкрийте файл рішення проекту (BoardGamesApp.sln) у середовищі Visual Studio 2022.

Переконайтеся, що проект збирається без помилок (клік правою кнопкою миші на проект -> Build).

Натисніть F5 або кнопку Start для запуску консольного застосунку.

Дотримуйтесь інструкцій у текстовому меню консолі для вибору гри та перегляду архітектурної симуляції.
